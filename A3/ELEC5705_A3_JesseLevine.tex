\section{Objective}
The objective of this assignment is to implement and verify the SAR logic for a 5-bit asynchronous SAR ADC operating at \SI{500}{MS/s} over a \SI{450}{mV} dynamic range. Assignment 2 focused on the comparator design. For Assignment 3, the emphasis is the control loop that performs the successive-approximation process and generates the next-bit decision timing without a fixed per-bit clock.

In this submission, the SAR logic is implemented and cross-checked using two open-source behavioral flows:
\begin{itemize}
    \item an Icarus Verilog/SystemVerilog simulation of the asynchronous controller,
    \item an \texttt{ngspice} behavioral experiment that reproduces the same successive-approximation process inside the \texttt{ngspice} control language,
    \item a Yosys gate-level synthesis of the SAR controller RTL.
\end{itemize}
In the behavioral verification flows, the comparator is modeled from the StrongARM latch transient results obtained in Assignment 2, rather than by reusing the original Cadence transistor-level implementation directly.

\section{Behavioral Model}
The ADC resolution is 5 bits, so the LSB is
\begin{equation}
\mathrm{LSB} = \frac{450\ \mathrm{mV}}{2^5} = 14.0625\ \mathrm{mV}.
\end{equation}

The simulated architecture consists of:
\begin{itemize}
    \item a 5-bit asynchronous SAR controller,
    \item an ideal 5-bit DAC,
    \item a StrongARM-inspired behavioral comparator,
    \item a sinusoidal input sampled at \SI{500}{MS/s}.
\end{itemize}

The controller begins each conversion by asserting the MSB trial code. After each comparator decision becomes valid, the logic either keeps or clears the current bit and immediately applies the next lower trial bit. This emulates asynchronous operation, where progression depends on comparator resolve time rather than a global bit clock.

\subsection{Comparator Model Derived from Assignment 2}
The comparator model is based on the StrongARM latch results from Assignment 2:
\begin{itemize}
    \item nominal decision time: approximately \SI{124}{ps},
    \item minimum decision time bound: \SI{60}{ps},
    \item maximum decision time bound: \SI{250}{ps},
    \item required minimum overdrive near $0.5$ LSB.
\end{itemize}

The behavioral delay is made overdrive-dependent so that larger $|V_{in} - V_{DAC}|$ produces a faster comparator decision, while smaller overdrive increases resolve time. This captures the key timing behavior of the StrongARM latch relevant to asynchronous SAR sequencing.

\section{Simulation Setup}
The testbench uses a near full-scale sinusoidal input centered at \SI{0.7}{V}, with a peak amplitude equal to $0.95 \times \frac{V_{FS}}{2}$. One conversion starts every \SI{2}{ns}, corresponding to the required \SI{500}{MS/s} rate.

The simulation produces:
\begin{itemize}
    \item the internal logic timing waveforms,
    \item the output codes for 1024 samples,
    \item the conversion-time distribution,
    \item an FFT-based SNDR and ENOB calculation,
    \item a gate-level digital realization of the SAR controller.
\end{itemize}

The Icarus flow models the SAR controller directly in SystemVerilog and drives it with a StrongARM-inspired comparator module. The \texttt{ngspice} flow does not use Verilog-A; instead, it implements the same 5-bit asynchronous approximation loop behaviorally in the \texttt{ngspice} control language using the same A2-derived comparator timing assumptions. The purpose of the second flow is to verify that the reported results are not an artifact of one simulator. In addition, the controller RTL was rewritten in a synthesis-oriented form and mapped to a gate-level netlist with Yosys as a structural digital cross-check.

\section{Cross-Check with ngspice}
To compare simulation frameworks, the same assignment was also implemented in a separate \texttt{ngspice} behavioral experiment. This version uses the same sinusoidal stimulus, the same DAC transfer function, and the same StrongARM-based comparator timing model from Assignment 2. Instead of instantiating a SystemVerilog SAR controller, the successive-approximation loop is executed in the \texttt{ngspice} control language and exported as waveform and code data.

Figure~\ref{fig:ngtiming} shows the \texttt{ngspice} timing trace over the same \SI{0}{ns} to \SI{6}{ns} multi-conversion window used for the Icarus waveform plot. The sequence of \texttt{start}, comparator-valid, trial code update, and end-of-conversion follows the same asynchronous logic behavior observed in the Icarus flow.

\section{Conclusion}
A 5-bit asynchronous SAR controller was implemented and verified using two open-source behavioral simulation flows and one gate-level synthesis flow. The comparator behavior was derived from the StrongARM latch designed in Assignment 2, allowing the SAR logic to be evaluated with a realistic decision-time model. Both the Icarus Verilog and \texttt{ngspice} methods produced essentially the same result, showing correct bit-cycling, asynchronous next-bit timing, and near-ideal 5-bit dynamic performance. The additional Yosys synthesis confirmed that the controller can also be mapped to a conventional digital gate-level netlist. The final reported performance is approximately \SI{31.12}{dB} SNDR and 4.88 bits ENOB, with sub-\SI{0.6}{ns} total conversion time.
